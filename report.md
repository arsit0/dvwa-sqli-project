# DVWA SQL Injection 实验报告

## 1. 实验目的
本实验基于 DVWA 搭建本地 Web 安全练习环境，通过 SQL Injection 模块完成基础注入复现，并对漏洞原理和防御思路进行总结。

## 2. 实验环境
- 操作系统：Windows
- Web 环境：XAMPP
- 靶场：DVWA
- 安全等级：low

## 3. 实验步骤与现象

### 3.1 正常查询
在 SQL Injection 模块中输入：

`1`

系统返回单个用户的信息，说明在正常情况下，输入内容被当作普通查询条件处理。

对应截图：
`screenshots/normal_query.png`

### 3.2 基础 SQL 注入
在输入框中输入：

`1' OR '1'='1`

页面返回了多条用户记录，而不再只是单个用户信息。这说明用户输入已经影响了后台数据库查询逻辑，原来的查询条件被绕过。

对应截图：
`screenshots/sql_injection_success.png`

### 3.3 信息提取型注入
在完成基础注入后，进一步尝试了信息提取型注入，页面显示出了更多敏感信息。说明 SQL 注入不仅可以改变查询逻辑，还可能进一步读取数据库中的关键数据。

对应截图：
`screenshots/data_extraction_union.png`

## 4. 漏洞原因分析
通过查看源码可以发现，程序将用户输入直接拼接进 SQL 查询语句，而没有使用参数化查询或预编译语句。这样一来，攻击者就可以通过构造特殊输入改变原本的 SQL 条件。

对应截图：
`screenshots/view_source.png`

我对漏洞成因的理解是：
SQL 注入的本质是程序把用户输入直接拼接进 SQL 查询语句，导致攻击者可以改变原本的查询条件，甚至进一步获取数据库中的敏感信息。

## 5. 风险分析
SQL 注入可能带来的风险包括：
- 泄露数据库中的敏感信息
- 绕过原有查询限制
- 获取本不应被访问的数据
- 严重时甚至可能影响数据库本身的数据内容

## 6. 防御思路
防御 SQL 注入的核心方法包括：
1. 不要直接拼接 SQL 语句
2. 使用参数化查询或预编译语句
3. 对输入进行基本校验
4. 控制数据库账户权限，避免权限过大

## 7. 实验总结
通过这次实验，我完成了 DVWA 本地环境搭建，并成功复现了 SQL 注入。相比只看概念，这种实际操作让我更直观地理解了 SQL 注入为什么会出现、会造成什么影响，以及为什么后端安全处理非常重要。
